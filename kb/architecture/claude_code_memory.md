# Oracle Forge — Claude Code Memory Architecture

## Context Engineering Design Notes

This document describes how Claude Code is used within Oracle Forge, and how the three-layer KB context is designed for reliable agent behavior.

---

## 1. Why Three Layers?

A single monolithic context file fails at scale:
- Too large → LLM ignores the middle (lost in the middle problem)
- Too generic → sub-agents get irrelevant context for their specific DB
- Not updatable → agent cannot learn from failures without a new deployment

Three layers solve these problems:

| Layer | Injected where | Update frequency | Who updates |
|---|---|---|---|
| Layer 1 (AGENT.md) | All sub-agents | Rarely (infrastructure changes) | Driver manually |
| Layer 2 (domain KB) | Conductor + matching sub-agent | After each schema introspection run | `utils/schema_introspector.py` |
| Layer 3 (corrections) | All sub-agents, end of prompt | After each benchmark run | `utils/autodream.py` |

---

## 2. Layer 1 Design Principles

`agent/AGENT.md` is the authoritative tool registry. It must be:
- **Verified**: every tool name, endpoint, and field comes from live inspection
- **Terse**: no prose — tables, bullet lists, exact examples only
- **Stable**: changes require team review (wrong tool name = silent failure)

Current content: 29 tools across 4 DB types, join key glossary, dialect rules per DB type, critical field notes (MongoDB has no `stars` or `city` field, etc.)

---

## 3. Layer 2 Design Principles

Domain KB files (`kb/domain/dab_*.md`) are generated by `utils/schema_introspector.py`:
1. Fetch schema from `GET /schema/{tool_name}` for each tool
2. Sample 10 rows from each table to confirm actual field values
3. Call Claude to enrich with domain context, field semantics, and known query patterns
4. Write to `kb/domain/dab_{dataset}.md`

Each file must pass injection testing before merge (see `kb/architecture/injection_tests/`).

---

## 4. Layer 3 Design Principles

Corrections log (`kb/corrections/corrections_log.md`) records:
```
[DATE] Dataset: {dataset} | Query: {query_id}
Wrong: {what the agent did}
Correct: {what it should do}
Impact: {what changed in KB or code}
```

`utils/autodream.py` reads the log, calls Claude to extract patterns, and appends a new `## Known Corrections` section to the matching domain KB file. This means Layer 3 gradually reduces Layer 2's blind spots.

---

## 5. Injection Testing Protocol

Before any KB document is committed to `main`, it must pass an injection test:
1. Craft a query that would fail without this KB document
2. Run the agent with the document injected (Layer 2 or 3)
3. Verify the agent produces the correct answer
4. Record the test in `kb/*/injection_tests/test_*.md`

Evidence format:
```
Query: [the test query]
Without KB: [what the agent did wrong]
With KB: [what the agent did correctly]
Result: PASS
```

---

## 6. Context Window Budget

Claude Sonnet 4.6 has a 200k token context window. KB context usage:
- Layer 1 (AGENT.md): ~2,000 tokens
- Layer 2 (one dataset's domain KB): ~1,000 tokens
- Layer 3 (corrections log): ~500 tokens
- Schema from GET /schema/: ~2,000 tokens
- Sub-agent prompt + prior results: ~3,000 tokens
- **Total per sub-agent call: ~8,500 tokens** — well within limits

Even with 10 sub-agent calls per query, total context per query is under 100k tokens.
